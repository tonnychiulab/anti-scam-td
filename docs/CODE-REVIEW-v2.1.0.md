## 一、結論摘要

v2.1.0 的核心玩法、87 關地圖生成、既有 PC 防護、版本一致性、四語字典結構、基本 PWA 快取與主要狀態機均通過現有測試。部署版能正常載入並進入遊戲，瀏覽器主控台沒有 error 或 warning；PC canvas 的右鍵不會被當成左鍵操作，右鍵選單防護仍有效。

本次沒有發現 P0 阻斷或 P1 高風險安全／資料損毀問題，但發現 **7 項 P2 中等問題、3 項 P3 低風險問題**。問題主要集中在手機手勢與面板狀態、暫停期間的技能蓄力、減少動態偏好、建塔無障礙、PWA 版本交接，以及少數已核准 UX 細節尚未落實。

因此，v2.1.0 可以正常遊玩，但目前不能判定為「D-01 至 D-10 全部無落差」或「已知零 bug」。建議以 v2.1.1 修補本報告的 P2 項目後，再做真機與螢幕閱讀器驗收。

## 三、詳細發現

### M-210-01｜手機手勢沒有完成建塔取消與單指平移

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:192` 要求開始雙指手勢時關閉建塔預覽且不扣點。
- 規格 `docs/UX-v2.1-mobile-guidance.md:211` 要求縮放後，單指拖動超過 tap-slop 時平移視圖。
- `game.js:2013-2026` 在第二個 pointer 加入時只呼叫 `cancelSupportAction(false)`，沒有呼叫 `closeBuildMenu()`。
- `game.js:2050-2052` 的單指拖動只把 `tapCand` 清空，完全沒有更新 `view.ox`、`view.oy` 或呼叫 `clampView()`。

**影響**

使用者開著建塔預覽縮放棋盤後，預覽仍留在畫面；之後可能在已改變視角的情況下誤確認舊格子。已縮放的棋盤也只能用雙指同時平移，與規格及一般行動裝置手勢習慣不一致。

**建議方向**

進入雙指手勢時關閉建塔預覽；另為單指拖曳保存起點與視圖偏移，在超過 tap-slop 且 `view.scale > 1` 時轉為 pan，不得送出 tap。

### M-210-02｜支援瞄準錯誤取消建塔預覽

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:252` 明定「進入瞄準不取消建造預覽；技能確認後才關閉其他面板」。
- `game.js:702-708` 的 `beginSupportAim()` 在剛進入瞄準時就執行 `closeBuildMenu()`。
- `game.js:724-731` 的全圖強光蓄力也立即關閉建塔面板。

**影響**

玩家建塔預覽到一半改看支援技能，只要進入瞄準，原本選定的塔種與格子就永久清除；即使取消支援也無法返回先前狀態，與已確認的操作流程相反。

**建議方向**

把建塔預覽與支援瞄準狀態分開保存；只有技能真正確認施放時才關閉其他面板，取消時恢復原建塔狀態。

### M-210-03｜強光蓄力在暫停中仍會完成

**證據**

- `game.js:724-738` 用原生 500ms `setTimeout()` 完成強光蓄力。
- timeout 回呼只檢查遊戲是否結束、蓄力 token 與場上敵人，沒有檢查 `S.paused` 或 `S.manualPaused`。
- `game.js:2278-2293` 開啟更多選單會暫停遊戲，但不會暫停或延後上述 timeout。

**可重現流程**

1. 在強光手電筒可用且場上有敵人時開始 500ms 蓄力。
2. 500ms 內按暫停或打開更多選單。
3. 保持暫停超過 500ms。
4. timeout 仍會呼叫 `useSupport(2)`，取消窗口消失並寫入技能冷卻；光束等遊戲迴圈恢復後才開始移動。

**影響**

「暫停」沒有凍結尚未完成的技能操作，玩家可能在查看設定時無意間消耗 50 秒冷卻。

**建議方向**

將蓄力改為由遊戲時間累積，或在進入暫停時保存剩餘時間並於恢復後續算；關閉更多選單前不得偷偷完成施放。

### M-210-04｜減少動態與最低品質仍震動

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:344` 指定減少動態模式的危險警示為靜態且無震動。
- 規格 `docs/UX-v2.1-mobile-guidance.md:367` 指定最低品質關閉粒子、震動與裝飾動畫。
- `game.js:429-437` 的危險警告直接呼叫 `navigator.vibrate(40)`，沒有檢查 `uxPrefs.reduceMotion` 或 `actualQuality === 'minimum'`。

**影響**

已主動選擇減少動態的玩家仍會收到觸覺刺激；自動品質降到最低時，也沒有得到規格承諾的完整保護。

**建議方向**

集中管理 haptic，至少在減少動態、最低品質與不支援震動時跳過；並新增對應回歸測試。

### M-210-05｜更多選單的偏好與內容不完整

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:320-321` 要求保存音效偏好，並在更多選單提供玩法說明。
- `game.js:144` 把 `muted` 固定初始化為 `false`。
- `game.js:158-184` 保存的 UX 偏好只有品質、減少動態、降低閃光與上次穩定品質，不含音效。
- `game.js:2252-2256` 切換靜音只改記憶體狀態，沒有寫入 `localStorage`。
- `index.html:195-216` 的更多選單只有分數、音效、品質、減少動態、降低閃光與返回；沒有玩法說明按鈕。

**影響**

重載、關閉分頁或重新開啟 PWA 後會恢復有聲；開始遊戲後也沒有入口重新查看操作教學。

**建議方向**

把音效納入既有 UX 偏好資料並安全遷移舊資料；在更多選單增加可開啟現有教學 dialog 的入口，返回時維持原暫停狀態。

### M-210-06｜建塔面板的無障礙與失敗回饋不足

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:388` 要求面板開啟後焦點進入面板，取消或完成後回到原棋盤格。
- `game.js:1839-1847` 只顯示建塔面板，沒有保存來源焦點或把焦點移入面板。
- `game.js:1875-1877` 完成後只更新 HUD 與 canvas，沒有回復棋盤格焦點。
- `game.js:1891-1899` 用 `.sel` 表示選中塔種，但動態按鈕沒有 `aria-pressed` 或其他選取語意。
- `index.html:121` 的取消鍵只有「✗」，沒有可讀的 `aria-label`。
- `game.js:1864-1865` 二次驗證失敗時直接關閉面板，沒有保留選擇或說明失敗原因。

**影響**

視覺使用者可完成建塔，但鍵盤與螢幕閱讀器使用者不會被帶到新面板、無法知道目前選了哪一塔，完成後也不知道焦點落點；驗證失敗時操作會無提示消失。

**建議方向**

保存來源棋盤格；開啟時聚焦面板標題或第一個可用塔；同步 `aria-pressed`；替取消鍵提供多語名稱；失敗時保留面板並以 live region 說明原因；完成／取消後回到來源格。

### M-210-07｜PWA 新版本首次載入仍可能使用舊版

**證據**

- `sw.js:72-83` 對所有同源 GET 採 stale-while-revalidate；只要舊快取存在就立即回傳舊內容。
- `sw.js:18-35` 雖然使用 `skipWaiting()` 與 `clients.claim()`，但只負責安裝與接管。
- `game.js:2639-2646` 只有註冊與快取統計，沒有監聽 `updatefound`、`controllerchange`，也沒有「新版本已就緒」提示或安全重載流程。

**影響**

發布 v2.1.1 等新版本後，已受舊 Service Worker 控制的玩家第一次進站可能先拿到舊的 HTML、JS 與 CSS，並在整個遊戲階段繼續使用舊版本；新 worker 接管後通常要再重新整理才看得到新版。

**建議方向**

導覽請求採 network-first，或在新 worker 接管後顯示更新提示並於安全時機重載；同時增加「舊 worker → 新 worker」的整合測試，避免只驗證快取名稱。

### L-210-08｜HUD 每個動畫幀重複操作 DOM

**證據**

- `game.js:888-1008` 的每個有效 animation frame 都呼叫 `updateHUD()`。
- `game.js:1747-1802` 每次都重新查詢 13 個塔按鈕及 3 個支援按鈕，並重寫文字、disabled、class、ARIA、波次與靜音狀態。
- D-08 的品質降級會減少視覺效果，但不會降低這段 DOM 查詢與寫入頻率。

**影響**

桌機通常感受不明顯，但低階手機在大量敵人與粒子同時出現時，這會增加主執行緒成本、耗電與掉幀風險。這是效能風險，本次沒有真機數據可量化成 P2。

**建議方向**

把 HUD 改成事件／dirty flag 更新；冷卻秒數等需定時更新的欄位可降到 4–10Hz，並快取固定 DOM 節點。

### L-210-09｜升級面板資訊不完整

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:199` 要求顯示下一級、費用與升級後餘額。
- `game.js:2121-2133` 只顯示目前塔名／等級，以及「升級費用 / 目前金額」。
- `game.js:2161-2171` 的 350ms 防連點及單級升級已正確實作。

**影響**

玩家能安全升級，但無法在按下前比較下一級傷害／射程，也必須自行計算升級後餘額。

**建議方向**

面板加入下一級傷害、射程或攻速差異，並直接顯示扣款後餘額；MAX 等級維持不可操作。

### L-210-10｜路線與危險導引的無障礙／多語缺口

**證據**

- 規格 `docs/UX-v2.1-mobile-guidance.md:386` 要求路線揭示後宣告「入口、出口與路線已顯示」。
- `game.js:338-345` 的 `startRouteGuide()` 只設定 canvas 動畫狀態，沒有送出 live-region 宣告。
- `index.html:105` 把 `#dangerEdge` 的 `aria-label` 寫死為中文「查看接近出口的敵人」。
- 實際部署版切換英文後，canvas 名稱會正確變成英文，但 `#dangerEdge` 的 accessible name 仍為上述中文，證實不是只有靜態碼上的理論缺口。

**影響**

非中文螢幕閱讀器使用者會遇到混合語言；純螢幕閱讀器玩家也無法得知新關卡的入口、出口與路線已完成顯示。

**建議方向**

新增四語路線完成與危險邊緣按鈕字串，於路線顯示時做一次 polite announcement，危險按鈕則隨數量與語系同步更新 accessible name。

## 七、建議修補順序與發行判定
